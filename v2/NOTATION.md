# esm-styles v2 — нотация

Зафиксирована 2026-07-23. Живая иллюстрация — воображаемое приложение в `v2/dream/`
(компилятора v2 ещё нет; спека первична, реализация подгоняется под неё).

## Принцип

Пара `компонент.tsx` + `компонент.styles.ts` — неделимая единица компиляции.
Компилятор **зрячий**: он читает разметку компонента и использует её
для компиляции точных селекторов и для диагностики.
Стиль — портрет разметки: вложенность ключей повторяет дерево элементов.

## Грамматика ключей

| Ключ | Смысл | Пример → CSS |
|---|---|---|
| имя тега | дочерний элемент | `header: { h1: {…} }` → `header > h1` |
| camelCase (не тег) | класс **на текущем элементе**, в CSS — kebab-case | `a: { active: {…} }` → `a.active`; `sideContent` → `.side-content` |
| PascalCase | граница дочернего компонента | `Byline: {…}` → `> .Byline` (точная форма селектора — решение компилятора) |
| `':…'`, `'::…'` | псевдокласс / псевдоэлемент на текущем | `':hover'`, `'::before'` |
| `'[…]'` | атрибут на текущем | `'[disabled]'` |
| `'@…'` | именованный media-запрос / тема из конфига | `'@mobile'`, `'@dark'` |
| `'.имя'` | класс, чьё имя совпадает с тегом (редкость) | `'.button'` |

Подчёркивания `_x` и `__x` — legacy v1. В v2 не нужны: у каждого класса
есть тег-хозяин (сдвиг вниз выражается самим деревом), коллизии решает `'.x'`.

## Корень

Корневой ключ экспорта — тег корневого элемента компонента:

```ts
export const Story = {
  article: { … },   // <article className="Story">
}
```

→ `article.Story`. Несовпадение тега с разметкой — ошибка.

## Пути и сокращения

Вложенность ключей — путь в дереве **собственной** разметки компонента.
Компилятор разворачивает его в точный селектор с `>`.
Промежуточные уровни можно опускать, если путь однозначен:

```ts
ul: { a: {…} }   // разметка ul > li > a  →  селектор ul > li > a
```

Неоднозначное сокращение (несколько путей) — предупреждение «уточни путь».
Несуществующий путь — ошибка. Поиск путей не заходит за PascalCase-границы.

## Границы компонентов

PascalCase-ключ можно позиционировать (`marginTop`, `order`, grid-размещение),
но нельзя заходить внутрь — внутренности принадлежат файлу самого компонента.
Родитель, желающий изменить вид ребёнка, просит вариант через prop
(`<Story compact>`), а вариант живёт в стилях ребёнка.

## Состояния и варианты

Обычные классы: camelCase-ключ на нужном элементе (`compact`, `urgent`,
`open`, `active`). Цепочка prop компонента ↔ класс в разметке ↔ ключ в стиле
проверяется зрячим инструментом.

## Зоны неизвестной разметки

Элемент, содержащий `{children}`, отрендеренный markdown или
`dangerouslySetInnerHTML`, переводит своё поддерево в свободный режим:
ключи компилируются обычными потомковыми селекторами, валидация выключена.
Режим определяется по разметке, а не синтаксисом стилей.

## Справочники `$`

Всё, что начинается с `$` — сгенерированный справочник (`$theme`, `$device`, …),
доступный как ambient-глобал: **импорты в файлах стилей не пишутся**.
Токен — обычная строка `'var(--theme-paper-bright)'`: работает значением,
в шаблонной строке, в конкатенации; никакого `.var`.
Значения по темам/устройствам — в JSDoc сгенерированного модуля (видны при hover).

## Типизация

Каждый экспорт подписывается:

```ts
export const Story = {
  …
} satisfies StyleOf<Markup.Story>
```

(глобальные стили — `satisfies GlobalStyle`). Namespace `Markup` — скелеты
разметки **в этой же нотации** (стиль без свойств), генерируются watcher'ом
в `esm-styles-env.d.ts` вместе с объявлениями `$`-глобалов и типов.
Файл окружения — целиком порождение генератора, руками не редактируется.

## Диагностика

- **Красное (tsc, через satisfies):** ключ вне скелета разметки; неверный
  корневой тег; несуществующее CSS-свойство или недопустимое значение.
- **Жёлтое (зрячий линтер):** элемент разметки без стиля; неоднозначный
  сокращённый путь; заход внутрь PascalCase-границы; класс в разметке,
  на который не ссылается ни один стиль.

## Этажи (floors)

Каскад организуется этажами, как в v1, но комплектование меняется:
этаж — не entry-файл с ручными импортами (их в v2 нет), а декларативная
группа, разрешаемая тем же резолвером, что находит пары:

```js
floors: [
  { name: 'global', source: 'global' },        // одиночный модуль
  { name: 'layout', include: 'layout.styles.ts' },
  { name: 'components', include: 'components/**' },
  { name: 'pages', include: 'pages/**' },
]
```

- Содержимое этажа — все экспорты всех пар группы. Коллизия имён
  экспортов внутри этажа — ошибка.
- Каждый этаж по умолчанию оборачивается в `@layer` со своим именем
  (отключаемо); порядок этажей = порядок каскада. `importFloors`,
  `outputPath`, `minify` — как в v1.
- Порядок файлов внутри этажа не имеет значения: границы компонентов
  запрещают взаимные переопределения по построению.
- Позиционирование ребёнка родителем живёт этажом выше ребёнка
  (pages после components) и выигрывает каскадом, а не специфичностью.

## Расположение файлов

Нотация не зависит от расположения. Пару определяет резолвер по конвенциям
из конфига — `components/story.tsx + styles/story.styles.ts`,
`components/story/index.tsx + styles/story/index.styles.ts`, co-location —
и по совпадению имени экспорта (`Story` ↔ `export const Story` ↔ `Markup.Story`).
Многофайловые компоненты допустимы: экстрактор скелета и так ходит
по импортам, чтобы распознавать границы компонентов.
